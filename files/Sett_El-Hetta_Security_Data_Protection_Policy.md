# SETT EL-HETTA — SECURITY & DATA PROTECTION POLICY

**Document Version:** 1.0 — Draft for Review
**Date:** July 19, 2026
**Companion documents:** Backend Implementation Plan v1.0 (§9) · SRS v1.0 (§5, NFRs) · Legal & Policy Documents Draft v1.0 (§2, Privacy Policy)

> The Backend Implementation Plan has a security section (§9) written for engineers implementing controls. The Privacy Policy draft has a security paragraph written for customers. Neither is a policy the business can actually operate against — what to do when something goes wrong, who's responsible for what, how long data is really kept, and what "incident response" means for a two- or three-person team with no dedicated security staff. This document is that operational layer, and it's also where the retention-period question — now raised in three prior documents (Backend Plan §12, Content Plan §7, Legal Docs §7) without being resolved — finally gets a recommended, concrete answer instead of being deferred a fourth time.

---

## 1. What Data This Platform Actually Handles

Plain inventory, not a technical schema — so this reads clearly to the business owner as well as the dev team:

| Data | Where it comes from | Sensitivity |
|---|---|---|
| Customer name, phone number | Checkout, contact form | Personal data |
| Delivery address | Checkout | Personal data |
| Optional email | Account registration | Personal data |
| Account password (hashed) | Account registration | Credential — never stored or transmitted in plain text |
| Vodafone Cash transfer-proof images | Checkout (uploaded by customer) | **Highest sensitivity** — can contain partial account numbers, transaction IDs, or other financial details visible in the screenshot itself |
| Admin login credentials | Admin accounts | Credential |
| Order history | Generated by the system | Personal + commercial data |

The payment-proof images are the single most sensitive thing this platform stores, and they're also the one piece of data unique to this project's specific payment model (Backend Plan §2) — most of the controls in this document exist specifically because of them.

## 2. Data Retention — Resolving the Open Question

This question has appeared, unresolved, in three prior documents. Recommending a concrete default here rather than leaving it open a fourth time:

| Data type | Recommended retention | Rationale |
|---|---|---|
| Payment-proof images | **18 months from order date** | Long enough to cover realistic dispute-resolution and basic accounting needs (SRS §5's stated requirement), short enough to limit exposure of financial-adjacent images sitting in storage indefinitely. |
| Order records (excluding proof images) | Retained indefinitely as standard business/sales records | Needed for reporting (FR-5.8), customer order history, and general accounting — no strong reason to delete unless the customer requests it. |
| Customer account data | Retained while the account is active; deleted on customer request per the Privacy Policy's stated rights (Legal Docs §2, Section 5) | |
| Admin action logs (`OrderStatusEvent`, `InventoryLog`) | Retained indefinitely | These are the audit trail the Backend Plan's Auditability requirement exists for — they should outlive the underlying order data if the two ever need to be handled differently. |

**This is a recommendation, not a final decision** — the business owner and/or legal reviewer (Legal Docs §6) should confirm or adjust the 18-month figure, but a concrete number should be locked in during Phase 1 (per Project Timeline §2) rather than remaining an open bracket through to launch, since the deletion mechanism (§3 below) can't be built without one.

## 3. Technical Controls (summary — full detail in Backend Plan §9)

- All traffic over HTTPS/TLS.
- Payment-proof images stored in a **private** bucket, never publicly listable, served only via short-lived signed URLs to authenticated admin users.
- Admin passwords hashed (bcrypt/argon2); JWT access tokens short-lived with revocable refresh tokens.
- Rate limiting on unauthenticated write endpoints (`POST /orders`, `POST /media/payment-proof`, `POST /contact`) to prevent abuse.
- Input validation on every public-facing endpoint.
- **New, added here:** a scheduled cleanup job that enforces the retention period in §2 automatically — deleting payment-proof images past 18 months rather than relying on someone remembering to do it manually. Manual deletion at scale is exactly the kind of task a time-constrained, non-technical owner (PRD §3) won't reliably do.

## 4. Access Control — Who Can See What

This restates the RBAC model from the Backend Plan and Design System Spec specifically through a security lens, since access control *is* a security control, not just a UX feature:

| Role | Can view payment-proof images? | Can view customer PII? | Can manage other admin accounts? |
|---|---|---|---|
| Owner | Yes | Yes | Yes |
| Staff | Yes | Yes | No |
| Delivery Coordinator | **No** | Limited (name, address, phone — needed for delivery only) | No |

**Enforcement note:** this needs to be enforced server-side (API-level guards, per Backend Plan §5.6's role column), not only hidden in the admin UI — a Delivery Coordinator account should get a `403` from the API if they attempt to fetch a payment-proof image directly, not just fail to see a button for it in the interface. This is already implied by the Backend Plan's design but is worth stating explicitly as a security requirement, not an implementation detail.

## 5. Incident Response — What to Actually Do

Written for a small team without dedicated security staff — deliberately short and actionable rather than a formal enterprise incident-response framework this project doesn't need.

**If you suspect unauthorized access to admin accounts or customer data:**
1. Immediately revoke/rotate the affected admin account's credentials and any active sessions (technically: invalidate refresh tokens in Redis, per Backend Plan §5.5).
2. Identify what data may have been exposed — check recent admin login activity and, if available, access logs for the payment-proof storage bucket.
3. Notify affected customers if personal data (especially payment-proof images) was likely accessed by an unauthorized party — this is both good practice and likely a legal obligation under Egyptian data-protection law (Law No. 151 of 2020); confirm specific notification obligations with the legal reviewer named in the Legal & Policy Documents plan.
4. Document what happened and what was done about it — even a simple written record is valuable if the incident needs to be discussed with a legal reviewer or, in a worse case, reported to a regulator.

**If a customer reports their payment-proof image or personal data was mishandled:**
1. Take the report seriously and acknowledge it promptly, even before investigation is complete.
2. Check the audit trail (`OrderStatusEvent`, admin access logs) to understand what actually happened.
3. Correct the issue and communicate directly with the customer about the outcome.

**Who owns this:** for a team this size, incident response is realistically the business owner's and the dev team's shared responsibility — there's no dedicated security officer, and this document shouldn't pretend otherwise. The dev team should be the first point of contact for anything technical (suspected breach, unusual access patterns); the business owner is the first point of contact for anything customer-facing (a customer reporting concern about their data).

## 6. Third-Party Data Sharing

Restating and slightly expanding the open item flagged in the Legal & Policy Documents draft (§2, Privacy Policy Section 4), specifically from a security standpoint:

| Third party | Data shared | Notes |
|---|---|---|
| WhatsApp Business API provider (if used) | Phone number, order status/content for notification messages | Acts as a data processor — confirm their own data-handling terms are acceptable before finalizing which provider is used (Backend Plan §12's open WhatsApp integration decision). |
| Email/SMS provider | Phone number/email, notification content | Same consideration as above. |
| Courier/delivery partner (Phase 4+, once integrated) | Name, phone, address — **never** payment-proof images | This boundary should be enforced by what data the courier-integration module is even given access to at the API level, not just a policy statement — worth flagging to the backend team when Phase 4 courier integration (Backend Plan §11) is actually built. |
| Hosting/cloud provider (AWS or equivalent) | All of the above, as the underlying infrastructure | Standard for any hosted application — no special action needed beyond choosing a reputable provider, already assumed in the Backend Plan's hosting recommendation. |

## 7. Relationship to the Privacy Policy

This document and the customer-facing Privacy Policy (Legal & Policy Documents §2) need to say the same thing about retention and data handling — once the 18-month figure in §2 above is confirmed, the Privacy Policy's bracketed retention placeholder should be updated to match exactly, not left as a vaguer "as required for accounting/dispute resolution" once a real number exists. A published policy that's vaguer than the actual internal practice is itself a small compliance risk.
